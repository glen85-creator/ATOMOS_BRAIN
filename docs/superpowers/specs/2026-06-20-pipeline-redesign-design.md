# ATOMOS 파이프라인 재설계 — 설계 문서 (LIVING / 진행 중)

> 상태: **코어 설계 완성** — 사용자 리뷰 → writing-plans 대기. (LIVING: 후속 결정 시 갱신.)
> 작성 시작: 2026-06-20. 대화로 합의된 결정을 누적 기록.

## 0. 배경 / 동기
- 기존 파이프라인(감지→외주→보고)이 개별 버그 땜질로는 해결 안 됨이 드러남(중복 차단·합성 flaky·발송 게이트 양방향 버그·토큰 분산·정량화 부족 등).
- **Paperclip(보드)·hermes(에이전트)는 Hostinger 관리형 이미지** = 우리 소스 아님, 휘발 패치만 가능한 블랙박스 → 의존이 문제의 근원.
- 결정: **전체 파이프라인을 우리가 통제하는 구조로 재설계.**

## 1. 범위 / 척추 결정 (확정)
- **범위**: 전체 파이프라인 재설계(감지→분석→승인→실행→측정→보고).
- **실행 모델**: **역할 단일단계** — 도메인별 역할 1개가 분석(다중 에이전트 팬아웃·CEO 합성 제거).
- **"실행" 정의**: **사람 승인 → 한정 안전도구 자동실행** + 효과 측정.
- **검증 실패 정책**: 분석 출력이 검증 게이트 실패 시 **거부→1~2회 재시도→그래도 실패면 "분석 실패"로 사람에게 표면화.** 쓰레기/trivial 출력 **절대 통과 금지**(오늘 "test" 발송 사고 방지).
- **1차 도메인 = 매장 매출(D-SALES)** — 매출 중심 데이터+분석 검증이 1차 핵심. **마케팅 대응(매출↓→캠페인)은 별도 트랙**(1차 코어 미포함).
- **에이전트 런타임**: **self-hosted Hermes (Nous Research)** 를 우리 VPS(Hostinger)에 직접 설치·통제. 관리형 Paperclip 보드 + 관리형 hermes는 **제거.**
- **실행 위치**: **(A) 엔진이 실행** — Hermes는 분석 + *읽기전용 조사*만, 사람 승인 뒤 **엔진이 한정 안전도구 실행.** (신뢰 쌓이면 추후 Hermes 실행(B)로 확장 검토.)

## 2. 타깃 아키텍처 (한 줄)
**우리 소유: [지식 wiki(매장/부서 스코프+권한) + 출력 검증 + 실행 안전(화이트리스트·승인·측정)]. Hermes = 우리가 스코프 컨텍스트를 먹이고, 묶인 도구로 (읽기전용) 조사·분석하는 런타임.**

## 3. 파이프라인 6단계
1. **감지(Detect)** — 기존 RPC/임계값(detection_settings) 유지. 이상치→안건(strategy_executions). evidence(수치) 표준화.
2. **분석(Analyze)** — 엔진이 도메인별 역할 프롬프트로 **self-hosted Hermes** 태스크 트리거. Hermes가 **우리 MCP 지식검색**으로 스코프 지식 회수 → 분석 → 구조화 출력(진단+제안액션+도구태그+정량효과). **출력 검증 게이트**(스키마·수치 정합·환각 차단)로 불량 거부·재시도.
3. **승인(Approve)** — 콘솔에서 운영자가 실행할 액션 선택·승인. 안건 식별 가능(title 표준화).
4. **실행(Execute)** — 승인 액션을 **화이트리스트 안전도구 레지스트리**(가역·멱등·스코프)로 엔진이 실행. before 스냅샷 자동.
5. **측정(Measure)** — 실행 전/후 지표 before→after 정량화(구별된 윈도).
6. **보고·발송(Report & Deliver)** — **단일 신뢰 게이트**(내용 품질·수신자·채널 검증)로 통과 후 등록 채널(이메일/메신저) 발송.

## 3a. 분석 단계 상세 (확정)
오늘 품질 사고(쓰레기·엉뚱한 전가·환각 수치)가 전부 여기서 났으므로 **검증 게이트가 핵심**.
- **입력**: 안건 + (MCP)스코프 지식 + **표준화 evidence**(수치·단위 일관).
- **역할 프롬프트**: 도메인→역할(1차: 매출→ANALYST). 페르소나 + 태스크 + 출력 스키마 + 주입 지식 + **그라운딩 지시**(준 데이터만 근거, 외부 추정 금지).
- **출력 스키마**: 진단 + 제안액션[각: title·what·how·owner·eta·**tool_tag**(실행 레지스트리 연결)·**정량 기대효과**] + risk + **confidence(통계 기반, 하드코딩 금지)**.
- **검증 게이트(거부→재시도→실패 표면화)**: ①스키마 유효 ②수치 정합(evidence 일치) ③그라운딩(실데이터 인용) ④환각 차단(근거 없는 수치 거부) ⑤trivial 거부.

## 3b. 매장 매출 감지·evidence 검증 (stage ①, 1차 핵심, 확정)
"매출 신호가 진짜 믿을 만한가"를 분석 전에 보장. 4 레이어:
- **L1 신선도·완전성**: 캐시(`sales_closing_monthly`) stale>26h·해당일 POS 동기 미완 → "데이터 미비" **분석 보류**.
- **L2 아티팩트 vs 진짜 급감**: `gross=0`(휴무/미수집)·defect-day(수집매장<윈도중앙값60%)·신규<90일·이력 n<5(conf=low) → 명시·투명 기록.
  - **L1/L2 미신뢰 → 별도 "데이터/운영 확인" 트랙** (AI 분석 큐 제외, 운영자 확인 요청). AI는 신뢰 신호만 분석.
- **L3 evidence 정합·표준화**: 단위 표준(전부 분수 저장·표기 시 ×100, `_pct` 버그 종결) / 전 필드 보존(sigma·n·dow·confidence None 폐기) / confidence = z-depth·표본·flag 산출(0.7 하드코딩 폐기).
- **L4 감지 정확도**: 주입 known-drop 픽스처로 RPC 단위검증(z경계 -2/-3·휴무·결함일) + 운영자 승인/반려 **라벨 축적 → precision/recall**(피드백 루프, 학습지식 v2 연결).

## 3c. 컨텍스트 데이터 레이어 (신뢰도/귀속, 확정)
외부 신호로 이상치를 설명/귀속 → 과잉반응 방지, 분석 신뢰도↑.
- **플러그형 프로바이더**: 매장(위치)+날짜 키로 정규화. 날씨(ASOS, *진행중*)·상권정보·캘린더(공휴일/행사)·…. **새 소스 = 프로바이더 1개 등록**(재설계 불필요 — "하나씩 늘림"과 일치).
- **주입 = 하이브리드**: 핵심(날씨·상권·캘린더)은 엔진이 evidence **자동 첨부** + 깊은 건 **MCP 컨텍스트 도구**(`context.weather`/`context.trade_area`)로 에이전트 읽기전용 pull.
- **활용**: 매출↓ ↔ 컨텍스트 **상관·귀속**(외부요인% vs 내부요인%) → confidence·권고에 반영(외부 크면 관망, 내부면 조치).
- 지식 wiki(큐레이션 markdown)와 구분: 컨텍스트 = 시계열/위치 **구조화 데이터**.

## 3d. 실행 안전도구 레지스트리 (stage ④, 매출 1차, 확정)
패턴: 도구 = `{id, params 스키마, 안전속성(가역·멱등·스코프), executor}`. 분석이 액션에 `tool_tag` 부여 → 사람 승인 → 엔진이 레지스트리로 실행.
매출 1차 도구(**가벼운 코어 — 데이터 변경 없음**; 매출 회복은 마케팅/사람):
- `notify(수신자,메시지)` · `create_task(담당,액션,기한)` · `record_decision(verdict,노트)` · `handoff_marketing(안건)` · `flag_data_issue(...)`(L1/L2 데이터-트랙).
- 공통 안전: 사람 승인 후 · 가역 · 멱등(중복 방지) · 스코프(매장/수신자 한정) · 실행 전 스냅샷(해당 시).

## 3e. 측정 (stage ⑤, 매출, 확정)
- **즉시(코어 통제)**: 진단 품질(운영자 승인/반려율·정확도 = L4 라벨) + 적시성(감지→알림→확인 지연).
- **지연(귀속 인식)**: 매출 회복(이상치 윈도 vs 후속 윈도 — **구별된 윈도**, 컨텍스트로 외부요인 보정). 마케팅 트랙과 **공유 효과** → 코어 단독 효과 과대주장 금지.
- **코어 KPI = 진단품질 + 적시성** / 매출회복 = 지연·귀속 추적(공유).

## 3f. 보고 신뢰게이트 (stage ⑥, 확정)
오늘 양방향 버그(쓰레기 통과·정상 fallback 차단) 교훈 → **내용 품질 기반 단일 게이트**:
- 체크: substance(trivial/placeholder/과도단축 거부) + grounding(evidence 수치 인용) + schema 완전성 + 수신자/채널 해소·유효 + 멱등(이미 발송). **`_fallback` 단독 의존 폐기.**
- **하나라도 실패 → 차단 + 사람 표면화**(쓰레기 발송 금지).
- 채널/수신자/토큰 **단일화**(VITE vs Railway vs 샌드박스 분산 종결 → 단일 설정·해소 함수).

## 4. Hermes ↔ 엔진 연동 계약 (확정 개념)
- Hermes는 **CLI/에이전트 우선, REST API 없음**(문서 확인). 트리거 = VPS SSH로 CLI 호출(비대화).
- **통합 단일 경계 = 우리 MCP 서버.** Hermes를 "우리 MCP 도구 + 최소 안전 내장도구"로만 화이트리스트(`hermes tools`/`hermes mcp`). `execute_code`·위험 도구 차단.
  - MCP 노출 도구: **(1) 스코프 지식검색**(권한·스코프 강제), **(2) 읽기전용 조사**(데이터 조회 등). *실행(쓰기) 도구는 엔진이 직접 — A안.*
- 보안: command-approval ON · 컨테이너 격리(Docker/Singularity) · 구조화 출력(파서/SGLang).
- 출처: hermes-agent.nousresearch.com/docs (mcp, security, cli-commands, configuration).

## 4b. 우리 MCP 서버 (단일 경계, 확정)
A모델: **읽기/조회 전용**(쓰기·실행 없음 — 실행은 엔진).
- 노출 도구: `knowledge.search(query,ctx)`(스코프·권한 검색) · `context.weather`/`context.trade_area`/`context.calendar`(컨텍스트 프로바이더) · `data.sales_history`/`data.store_profile`(읽기·스코프).
- **사전정의 안전 도구만**(raw SQL/임의 접근 없음). **확장 가능** — 새 읽기/조회 도구는 레지스트리 등록(컨텍스트 프로바이더처럼). "사전정의"=*우리가 큐레이션*, 상한 아님; 언제든 추가.
- 스코프·권한은 **MCP 서버가 호출 ctx로 강제**(Hermes 우회 불가).
- **호스팅 = 우리 엔진(Railway) 측**; Hermes(VPS)가 원격 MCP로 연결(DB·권한과 동일 위치, 일관).
- Hermes 화이트리스트 = 이 도구들 + 최소 안전 내장도구.

## 5. 지식 wiki 레이어 (확정)
- **소스 = GitHub `ATOMOS_BRAIN/knowledge/`** (마크다운, 폴더=스코프). VPS 아님(휘발·권한우회 회피).
- **색인 = Supabase pgvector** (임베딩 + 메타: scope/dept/store/read_scope/source/verified). git push → CI/sync 잡으로 색인.
- **검색/주입 = 우리 MCP `knowledge.search(ctx)`** — 호출 역할+안건 기준 스코프·권한 필터 통과분만 반환. Hermes는 지식을 보유 안 함.
- **스코프**: `global` / `brand:{br_id}` / `dept:{SCM|FINANCE|…}` / `store:{st_id}`.
- **권한 2층**: GitHub 협업자(누가 *작성*) + MCP 런타임(각 역할/안건이 *읽을* 네임스페이스).
- **지식 2종**: 참조지식(사람 큐레이션) — **v1 채택**. 학습지식(검증된 사이클 자동축적) — **v2로 연기**.
- **품질 가드**: 학습은 검증된 사이클만 / 사람 "틀림" 피드백 / 출처·신뢰도 메타.

## 6. 가로지르는 기반 (오늘 거친 seam 구조적 제거)
- 단일 상태원천(DB 한 곳) → 멱등·중복 분열 제거.
- 관측성: 모든 단계 동기·기록·실패 표면화, 실비용(추정 폐기).
- 단일 트리거/멱등 → 중복 합성·반복 호출 불가.
- 설정 단일화(토큰·플래그·채널 한 곳).

## 8. 구현 범위 / 1차 도메인
아키텍처는 범용. **1차 핵심 = 매장 매출(D-SALES) 중심 데이터+분석 검증** (가장 중요한 비즈니스 신호). 역할 = 매출 분석가(ANALYST). **마케팅 대응 줄기(매출↓→캠페인)는 별도 트랙으로 분리**(1차 코어 미포함, 후속). 이후 COGS·비용·리뷰 도메인 확장.
> D-COGS는 *구 시스템 사이클 테스트*에 썼던 슬라이스 — **재설계 1차는 매출(D-SALES).**

## 10. 구현 단계 (phasing) — 키스톤 우선
- **Phase 1 (키스톤): self-hosted Hermes VPS 셋업 + 우리 MCP 서버.** 가장 큰 미검증 가정 → 최우선 de-risk.
  - 설치: Hostinger VPS에 self-hosted Hermes(shell 설치, Docker 백엔드 권장=격리). 관리형 `hvps-hermes`/`hvps-paperclip`과 분리된 우리 인스턴스.
  - 설정: 모델/프로바이더(`hermes setup`) · 도구 화이트리스트(`hermes tools` — `execute_code` 등 위험 off) · MCP 연결(`hermes mcp` — 우리 MCP: `knowledge.search` + `context.*` + read-only 조사) · 보안(command-approval ON·컨테이너 격리) · 구조화 출력 파서.
  - 트리거 배선: 엔진(Railway) → VPS Hermes 태스크(SSH CLI 비대화) → 구조화 출력 회수.
  - **PoC 합격선**: 매출 안건 1건이 MCP로 스코프 지식 받고 → 분석 → 구조화 출력 회수까지 e2e. "우리가 Hermes를 통제 구동 가능" 입증.
- **Phase 2: 매출 도메인 수직 슬라이스** — §3b 검증 + §3c 컨텍스트 + §3a 분석 + §3d 실행 + 측정 + 보고게이트, 매출 1건 e2e.
